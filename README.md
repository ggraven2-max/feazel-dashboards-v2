## Daily workflow

### Option A: terminal (fastest if you're at your machine)

```bash
# 1. Drop today's CSV / XLSX exports in inputs/<lob>/<project>/
cp ~/Downloads/Contracts\ Signed\ YTD\ -\ Residential-2026-05-07.xlsx \
   inputs/residential/sales-overview/

# Multi-Family and Service follow the same pattern:
cp ~/Downloads/Commercial\ Forecasting\ Report-2026-05-07.xlsx \
   inputs/multi-family/revenue-forecast/

# 2. Run the build (all LOBs, all projects)
npm run build

# Or scope to one LOB / project:
npm run build -- --lob residential
npm run build -- --lob residential --project sales-overview

# 3. Open any dashboard tab in your browser, refresh
open redesign/index.html
```

### Option B: drag-and-drop (no terminal needed)

1. Open `admin/upload.html` in your browser
2. Drag your CSV/XLSX exports into the appropriate project zones
3. Click **Process & Download data.js**
4. Drop the downloaded `data.js` into `redesign/shared/data.js`
5. Refresh any dashboard tab

### Option C: scheduled (steady state)

After Ben's team wires up Salesforce report exports to a folder watch (or SFTP drop), `npm run build` runs nightly via cron / Task Scheduler / GitHub Actions. The Netlify site auto-publishes on git push of the new `data.js`. No manual step.

## How rules get added (the important part)

Each project has a `RULES.md` file in `inputs/<lob>/<project>/`. Open it to see the current rules and add new ones. Each rule gets a number (`RULE-NNN`), a one-line definition, and an approver.

When you have a rule documented, encode it in the matching `calculators/<project>.js` file in the marked **IMPLEMENT_RULES** block. Replace the `readFromExtracted()` passthrough with real CSV parsing + aggregation. Use the helpers in `calculators/lib/aggregate.js` (`groupBy`, `sumBy`, `median`, etc).

When the rule is in code, run:

```bash
node calculators/tests/run.js -- --update
```

This locks the current output as the snapshot. From then on, any change to the calculator that produces different numbers will fail the test until you intentionally re-snapshot with sign-off.

## Methodology lock (V5)

The Residential Revenue Forecast calculator is a shell wrapper over the locked Python implementation (`refresh_v5.py`). The Python is the source of truth; the JS calculator just stages inputs and parses the output.

Lock state (as of 2026-05-04, see `FORECASTING_RULES.md` Section 4 for the full table):

| Element                             | State                  | Lock date  |
|---|---|---|
| WIP percent-complete = 0.75         | LOCKED                 | 2026-04-19 |
| WIP reference baseline (`wip_reference.pkl`) | LOCKED        | 2026-04-19 |
| Branch consolidation (NOVA вҶ’ DC Metro)       | LOCKED        | 2026-04-19 |
| Annual budget anchor                | LOCKED                 | 2026-04-19 |
| Pipeline ownership (`build_refresh_v2.py` canonical) | LOCKED | 2026-04-30 |
| NetSuite invoicing source           | LOCKED                 | 2026-05-04 |
| Closed-month auto-detection         | LOCKED                 | 2026-05-04 |
| Cycle time hierarchy                | UNLOCKED, recomputed each refresh | 2026-04-30 |
| Same-month conversion               | UNLOCKED, recomputed each refresh | 2026-04-30 |
| MMU (material mark-up)              | REMOVED                | 2026-04-30 |

The `V5_CONSTANTS` block in `calculators/revenue-forecast.js` carries the **stale defaults from the 2026-04-19 lock** for reference only (cycle and conversion arrays are tagged `_stale_defaults`). The Python recomputes the unlocked elements every refresh and is authoritative.

Changes to anything still LOCKED require:
1. Written sign-off (Greg + Mahlet)
2. PR with new `V_NEXT` version label and updated `lockHistory` table
3. Updated snapshot in `calculators/tests/snapshots/`
4. Entry in the change log in `inputs/residential/revenue-forecast/RULES.md`

Multi-Family uses its own `MF-v1` model (event-driven, not statistical) and Service uses `Service-v1` (annualized run-rate). Each carries its own lock table in the matching `inputs/<lob>/revenue-forecast/RULES.md` file.

## Deploy

The web suite deploys to Netlify from this repo. See `netlify.toml`. Every push to `main` rebuilds and publishes. The iOS app is a separate (Expo) build, see `App.js` and run `npm run ios`.

## Watch list

- **Salesforce report drift.** The day someone in Sales Ops renames a column, the calculator will fail with a clear error. Pipe build failures to a Slack channel so we catch it the same day.
- **Concentration risk.** Don't let this live in only one person's head. The RULES.md files exist so any developer can pick this up cold.
- **Data privacy.** Raw uploaded CSVs in `inputs/` are gitignored by default. Don't commit customer data.

## Quick reference

| Command | What it does |
|---|---|
| `npm run build` | Run every calculator across every LOB, write fresh per-LOB data.js + combined data.json |
| `npm run build -- --lob residential` | Run one LOB only (residential, multi-family, or service) |
| `npm run build -- --lob residential --project sales-overview` | Run one project under one LOB |
| `npm run validate` | Run calculators but don't write output (dry run) |
| `npm run test:calculators` | Snapshot tests, fail on methodology drift |
| `npm run test:calculators -- --update` | Re-snapshot the current outputs (use after an intentional methodology change) |
| `npm run serve` | Local HTTP server on port 8765 to view the dashboards |
| `npm run ios` | Launch the Expo iOS app |
